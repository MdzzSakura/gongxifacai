# 稳定数据获取模块 — 架构设计（深化版）

生成时间：2026-07-07
状态：待评审
前置设计：[2026-06-30 本地数据落地：采集/筛选解耦（DuckDB）](2026-06-30-local-data-store-ingest-screen-design.md) — 本篇不推翻其结论（DuckDB 单文件、严格离线筛选、ingest/screen 分离），只深化其中"采集侧如何稳定拉数据"这一层。
参考实现：Sequoia-X（baostock + SQLite 增量）、myhhub/stock（Session 重试 + 本地缓存）、aiagents-stock（多源降级 + 指数退避）、KHunter（除权检测 + 智能更新时机）。

---

## 1. 目标与非目标

**目标**

1. 盘后一次采集，把当日全部所需数据增量落进 DuckDB；策略全程只读本地。
2. 全市场日K 的**每日增量成本从 ~5000 次请求降到 1~2 次**（快照一次成型）。
3. 任何单一免费源被风控/宕机，采集自动换源完成，不需要人工介入。
4. 除权导致的前复权历史失真能被**自动检测并自愈**。
5. 每轮采集留下结构化台账（采了什么、用了哪个源、失败了什么），可断点续传、可审计。

**非目标**

- 盘中实时监控（快照接口能力保留，但不在本设计范围）。
- 分钟线、Level-2、资金流向等扩展数据集（表结构预留扩展方式即可）。
- 代理池/Cookie 伪装（myhhub 路线维护成本高，我们靠"降低请求量 + 换源"解决，明确不做）。

## 2. 总体架构

```
                    ┌──────────────────────────────────────────┐
                    │ ingest.py  采集编排（唯一联网入口）          │
                    │  交易日历 → 算缺口 → 按数据集逐一补齐         │
                    └───────┬──────────────────────┬───────────┘
                            │                      │
                ┌───────────▼──────────┐  ┌────────▼─────────┐
                │ Fetcher（多源路由）    │  │ QualityGate       │
                │  源注册表 + 优先级      │  │  入库前校验闸门     │
                │  节流/退避/熔断/健康度  │  └────────┬─────────┘
                └───────────┬──────────┘           │
              ┌─────────────┼─────────────┐        │
        ┌─────▼────┐  ┌─────▼────┐  ┌─────▼────┐   │
        │ baostock │  │  新浪     │  │  东财     │   │  （腾讯：日K第四路，预留）
        └──────────┘  └──────────┘  └──────────┘   │
                                                   │
                    ┌──────────────────────────────▼───────────┐
                    │ DuckStore（gxfc_data.duckdb）              │
                    │  业务表 + trade_calendar + ingest_log      │
                    └───────────────────┬───────────────────────┘
                                        │ 只读
                              ┌─────────▼─────────┐
                              │ screen.py 离线筛选  │
                              └───────────────────┘
```

分层职责：

| 层 | 职责 | 归属文件 |
|----|------|---------|
| 源适配 | 每个源的取数 + 归一化为东财列结构（现有 `_sina_*`/`_baostock_*` 函数群） | `gxfc/data/fetcher.py` |
| 多源路由 | 优先级回退、每源节流参数、熔断、健康度计数（演进现有 `_fetch_first_ok`） | `gxfc/data/fetcher.py` |
| 采集编排 | 交易日历、缺口计算、增量策略、除权自愈、失败预算、台账 | `gxfc/ingest.py`（新） |
| 质量闸门 | 空表/缺列/行数骤降拦截，坏数据不入库 | `gxfc/data/quality.py`（新） |
| 存储 | DuckDB 读写、upsert/append 去重、水位查询 | `gxfc/store/duck_store.py` |

## 3. 核心设计

### 3.1 交易日历是增量的骨架

一切"该不该采、缺哪几天"的判断都以本地 `trade_calendar` 表为准，不再用"今天是不是周末"之类的猜测。

- 来源：baostock `query_trade_dates`（自有服务器，最稳），每次采集开头刷新"上次日历截止日 → 今天"这一小段；失败则回退 akshare `tool_trade_date_hist_sina`。
- 缺口计算：`缺口 = calendar 中 (水位线, 最新已收盘交易日] 的交易日集合`。
- 17:30 前跑采集时，baostock 当日数据可能未出：日K增量走快照源（盘后即有），baostock 仅用于历史回补，两者时效性正好互补。

### 3.2 两类数据集、两种增量语义

| 类型 | 数据集 | 增量语义 | 续传判断 |
|------|--------|---------|---------|
| 快照类 | zt/dt/zb 池、板块榜、板块成分、业绩预告 | 按期 upsert（先删该期再插） | `ingest_log` 中该 (dataset, period) 已 `ok` 则跳过 |
| 序列类 | 日K `daily` | 按 (代码,日期) append 去重 | 表内 `MAX(日期) per 代码` 对照交易日历 |

**为什么快照类不用 `snapshot_dates` 而用 `ingest_log`**：涨停池某天可能真的为空（极端行情），"表里没有该日数据"区分不了"没采过"和"采过但为空"，会造成每次重跑都空采一遍。台账记录"采过、0 行、成功"即可幂等跳过。

**为什么日K不引入独立水位线表**：`MAX(日期)` 就是单一事实来源，独立水位表会和数据漂移（Sequoia-X 同款选择）。停牌股不会因此每天被重拉——见 3.3，每日增量根本不逐股发请求。

### 3.3 当日日K"快照一次成型"（本设计最大的降载点）

全市场约 5400 只股票的**当日**日K，不逐股拉，而是收盘后拉一次全市场快照，直接转写为当日 daily 行：

```
东财 spot_em（1次请求，含 今开/最高/最低/最新价/成交量/成交额/换手率/昨收）
  → 熔断/失败 → 新浪 stock_zh_a_spot（1次请求，无换手率，该列置 NULL）
  → 转为 daily 表行（trade_date = 当日），append 去重入库
```

- 每日常态请求数：日历 1 次 + 快照 1 次 + 快照类数据集约 6~15 次 ≈ **20 次以内**，任何免费源都远够不到限流阈值。这比"伪装+代理"从根上解决稳定性。
- 逐股拉取（baostock → 新浪 → 东财，即现有 `stock_daily` 三路回退）只剩两个场景：**新股/缺口历史回补**、**除权重拉**（3.4）。回补量大时靠断点续传分次完成。

### 3.4 除权检测与自愈（吸收 KHunter 经验）

库里存前复权价（现有因子如 `profit_fault.detect_gap` 依赖前复权连续性，不动它们）。前复权与增量追加天然矛盾：某票除权后，它的**全部历史前复权价都变了**，只 append 当日行会得到断裂序列。

处理：快照自带"昨收"字段（交易所口径，除权日会变成除权后价格），入库前逐票比对：

```
快照.昨收 ≈ 库内该票最后一日.收盘  → 无除权，正常 append
否则                              → 记入除权名单，删该票全部历史，
                                    重拉前复权全窗口（走 stock_daily 三路回退）
```

- 每日代价：一次向量化比对（零请求）；重拉量 = 当日除权股数（常态几只到几十只）。
- 容差：`abs(diff)/昨收 > 0.5%` 判为除权（浮点与四舍五入留余量）。
- 除权名单写入 `ingest_log`，重拉失败的票次日自动重试（因为其 `MAX(日期)` 落后，会被当作缺口）。

### 3.5 多源路由：演进现有 `_fetch_first_ok`，不重写

现有机制保留：优先级列表、东财只试 1 次、连接被掐即熔断整轮、最后兜底源用满重试预算。在此之上补三点：

1. **源注册表**：把散在各业务方法里的 `[("东财", loader, True), ...]` 收敛为模块级 `SOURCES: dict[dataset, list[SourceSpec]]`，`SourceSpec = (name, loader工厂, is_eastmoney, min_interval)`。新增/调整源只改注册表。
2. **每源独立节流**：baostock 自有服务器可 0.1s 间隔；东财维持 0.8s+ 并加 ±30% 随机抖动（等间隔请求是最容易被指纹识别的爬虫特征）。`_throttle` 从实例级单一间隔改为按源计时。
3. **健康度计数**：每源记录 `(成功数, 失败数, 连续失败数)`，采集结束写入台账摘要。连续失败 ≥3 的非东财源也临时熔断本轮（现有熔断只覆盖东财）。

日K逐股回退链预留第四路**腾讯**（`stock_zh_a_hist_tx`，aiagents-stock 被东财封 IP 后的主力源），实现放在注册表里加一行即可，首版可不接。

### 3.6 失败预算：快速失败优于带病空转

网络整体不可用时，逐股回补会把 5000 只票 × 三源 × 重试全部烧完（几小时的无效重试）。规则：

- **连续 20 只**股票所有源全失败 → 判定网络/源整体故障，中止本轮采集，台账记 `aborted`，退出码非 0。
- 已入库的数据不回滚（断点续传保证下次接着采）。
- 单票失败（个股停牌、退市、数据缺失)不计入连续失败？——计入但成功即清零；停牌票不会连续扎堆 20 只，误判概率可忽略。

### 3.7 质量闸门：坏数据不入库

`quality.py` 提供 `validate(dataset, df) -> df`，在写 DuckStore 之前统一过闸：

| 检查 | 规则 | 动作 |
|------|------|------|
| 必需列 | 每个数据集声明必需列集合（如 daily：日期/开盘/最高/收盘） | 缺列 → 抛错，该数据集本轮失败 |
| 空表 | 快照类空表 | 允许入台账（0 行），但打 warning |
| 行数骤降 | 全市场快照行数 < 上一交易日的 80% | 拒写 + 报错（半残数据比没有更危险） |
| 数值粗检 | 涨跌幅 abs > 45%、价格 ≤ 0 | 剔除该行 + warning 计数 |
| 去重 | (代码,日期) / (期, 代码) 重复 | 保留最后一条 |

### 3.8 可观测性：ingest_log 台账

```sql
ingest_log(
  run_id      TEXT,      -- 一次 ingest 进程一个 id（时间戳）
  dataset     TEXT,      -- 'daily_snapshot' / 'zt_pool' / 'daily_backfill:600519' ...
  period      TEXT,      -- trade_date 或 quarter_end
  status      TEXT,      -- ok / empty / failed / aborted
  rows        INTEGER,
  source      TEXT,      -- 实际成功的源名
  error       TEXT,
  finished_at TIMESTAMP
)
```

- 采集结束打印一屏摘要：各数据集状态、各源成功/失败计数、除权重拉名单、耗时。
- `screen.py` 开头查台账：目标交易日数据集不全 → 面板标注"某段未采集"（沿用既有分段降级）。

## 4. DuckDB 表结构（在 06-30 设计上追加两张系统表）

业务表（zt_pool / dt_pool / zb_pool / industry_board / industry_cons / yjyg / daily）沿用 06-30 设计不变，追加：

```sql
trade_calendar(calendar_date DATE PRIMARY KEY, is_trading_day BOOLEAN)
ingest_log(...)   -- 见 3.8
```

daily 表明确列（东财中文列口径，全源归一后统一）：

```sql
daily("代码" TEXT, "日期" TEXT, "开盘" DOUBLE, "收盘" DOUBLE, "最高" DOUBLE,
      "最低" DOUBLE, "成交量" DOUBLE, "成交额" DOUBLE, "换手率" DOUBLE,
      PRIMARY KEY ("代码", "日期"))
```

（新浪快照源无换手率 → NULL；因子侧消费该列前需判空。）

## 5. 采集主流程

```
run_ingest(target_date=None):
  1. 刷新 trade_calendar（baostock → 新浪回退）
  2. target_date 默认取最新已收盘交易日；非交易日直接退出
  3. 快照类数据集（zt/dt/zb/板块榜/成分/业绩预告）:
       for ds in snapshot_datasets:
         if ingest_log 已 ok(ds, period): skip
         df = fetcher.<ds>()  → quality.validate → store.upsert_snapshot → 记台账
  4. 当日日K（快照一次成型）:
       spot = fetcher.spot()（东财→新浪回退）
       除权检测：spot.昨收 vs daily 表各票最后收盘   (3.4)
       正常票 → 转 daily 行 append；除权票 → 删历史，进回补队列
  5. 历史回补（新股/除权/往日缺口）:
       codes = daily 表 MAX(日期) 落后于日历的票 ∪ 除权名单 ∪ 新上市票
       for code in codes:  （断点续传：每票独立，中断重跑接着采）
         fetcher.stock_daily(code, 窗口起点, target_date)  # baostock→新浪→东财
         → validate → append → 台账
         连续 20 票全源失败 → abort            (3.6)
  6. 打印采集摘要
```

CLI 保持 06-30 设计：`python -m gxfc.ingest [date]`（quarter_end 由 date 推导，不再要求手工传）。

| 步骤 | 内容 | 验证 |

## 7. 测试策略

- **单测**（不联网）：DuckStore 幂等性/去重、缺口计算、除权判定（构造昨收偏移样本）、质量闸门各规则、失败预算中止、多源回退顺序与熔断（FakeSource 抛 `RemoteDisconnected`）。
- **冒烟**（联网，手动触发）：真实跑一次 `ingest` 小样本（限定 20 只票），断言库内行数与台账一致。
- **离线验证**：断网状态跑 `screen`，确认零网络请求且降级标注正确。

## 8. 风险与对策

| 风险 | 对策 |
|------|------|
| 快照 OHLC 与日K接口口径有微小差异（集合竞价处理等） | 除权检测顺带校验昨收一致性；发现系统性偏差再切"当日也逐股 baostock"（17:30 后跑） |
| 新浪快照无换手率 | 列置 NULL，消费方判空；东财恢复后自然补齐后续日期 |
| baostock 服务偶发维护 | 日K三路回退已覆盖；日历回退新浪 |
| 免费源接口列名变更（akshare 升级） | 归一化层集中在 fetcher 一处；quality 缺列检查会在入库前拦截并指明数据集 |
| 首次全市场回补耗时（小时级） | 断点续传 + 进度日志；建议首采分 2~3 晚完成 |
